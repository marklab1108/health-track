# 後續 Phase Roadmap：Health Track

依照先前 `/office-hours`、`/plan-eng-review`、`/plan-design-review` 與整合 review 的結論整理。

這份文件不是重新發明需求，而是把「MVP 之後該做什麼」收斂成可執行的後續 phases。

## 目前 Phase 1 已完成的核心

目前第一版已經完成的核心回饋迴路是：

1. 個人化目標設定
2. 今日儀表板
3. 快速記餐
4. 常吃餐點模板
5. 體重輸入
6. 週 / 月趨勢回顧
7. Local-first PWA-lite
8. Demo data、基本測試與部署基礎

所以後續 phase 的目標，不是把同一件事做更大，而是補上真正能提升使用價值的下一層能力。

## Review 結論

### P1：後續 phase 不應該先做的事

以下項目雖然常見，但不該優先進入下一階段：

- 完整食物資料庫
- 條碼掃描
- 社群功能
- 食譜 / 餐點推薦
- 原生 iOS / Android app
- App Store / Play Store 發佈
- 醫療級建議

原因：
- 這些都會快速擴大 scope，但不會優先強化你目前最有差異化的產品價值。
- 你的核心仍然是「低摩擦記錄 + 飲食與體重的回饋迴路」。

### P1：最值得優先補的方向

真正應該進入下一個 phase 的，是這四類：

1. 更低摩擦的記錄流程
2. 更聰明的回顧與建議
3. 長期使用所需的同步與平台延伸
4. 更後期的隱私升級

---

## Phase 2：讓第一版真的可長期使用

目標：補齊現在最明顯的使用缺口，讓 app 不只是能 demo，而是可以連續用幾週。

### 建議功能

1. 強化記餐速度
   - 複製昨天餐點
   - 複製最近一筆餐點
   - 模板排序：最近使用 / 最常使用
   - 今天頁直接快速新增常吃餐點

2. 強化回顧可讀性
   - 記錄完整度提示
   - 缺少資料時的保守提示
   - 蛋白質是否明顯不足
   - 熱量與體重方向是否一致

3. 目標與歷史
   - 保留目標變更歷史
   - 回顧時顯示該期間使用的是哪組目標

4. 使用體驗收斂
   - 更清楚的資料狀態提示
   - 空狀態與錯誤狀態優化
   - 更穩定的 demo / 測試資料流程

### 為什麼這是 Phase 2

- 快速複製與模板優化能直接提升「30 秒記一餐」這個核心承諾。
- 回顧可讀性強化，能把現有圖表變成更實用的判斷工具。
- 這一階段先不做資料可攜性，而是先把日常使用體驗補齊。

### 完成標準

- 記錄常吃餐點不需要每次重填
- 回顧畫面不只顯示數字，也能判斷「資料夠不夠看」
- 使用者可以連續用幾週，不會被操作摩擦打斷

---

## Phase 3：把 app 從 tracker 推進到 coach

目標：從單純顯示結果，進一步提供保守但有用的調整建議。

### 建議功能

1. `adaptive target`
   - 根據數週體重趨勢調整下週熱量建議
   - 維持保守規則，不做過度自信推論

2. `weekly insight v2`
   - 整合熱量偏差、蛋白質、記錄完整度、體重方向
   - 產出更具體的下週建議

3. `自然語言週摘要`
   - 例如：
     - 這週平均攝取接近建議量
     - 體重比上月同期下降
     - 蛋白質穩定，但高熱量餐集中在週末

4. `目標微調入口`
   - 使用者可接受或忽略系統建議
   - 建議不直接覆蓋目標，而是先顯示建議值

### 為什麼這是 Phase 3

- 這是你產品和一般 calorie tracker 最可能拉開差異的地方。
- 但它需要先有足夠歷史資料、足夠穩定的記錄習慣，否則很容易變成亂猜。

### 完成標準

- 使用者看完回顧後，知道下週要不要維持、減少或增加攝取
- 建議文案保持保守，不像醫療建議

---

## Phase 4：降低記錄摩擦的 10x 功能

目標：把輸入速度再往前推一級，但只在核心 loop 已證明有用後才做。

### 建議功能

1. 語音記餐
   - 口語輸入餐點與營養資訊
   - 轉成結構化表單後再讓使用者確認

2. 照片記餐
   - 拍照後產生初始估算
   - 使用者手動確認 / 修正

3. 長按快速新增
   - 從今天頁或模板列表快速插入常用餐點

4. 更好的餐點搜尋
   - 不是大型公共資料庫
   - 先從自己的模板與歷史餐點搜尋開始

### 為什麼這是 Phase 4

- 語音和照片很吸引人，但它們不是第一個瓶頸。
- 如果沒有先證明「使用者真的會回來看回顧」，再快的輸入也只是在優化一個未被證明的流程。

---

## Phase 5：跨裝置與平台延伸

目標：先支撐「自己和朋友使用」的同步版本，但只做到最低必要安全底線。

### 建議功能

1. 帳號系統
2. 雲端同步
3. 多裝置一致性
4. 衝突處理策略
5. 資料 migration
6. Native mobile 評估
7. 最低安全底線
   - 明確知道資料存在本機還是雲端
   - 最小蒐集
   - 基本 auth / 權限隔離
   - 可刪除資料
   - 不把敏感資料偷偷送第三方

### 為什麼這是 Phase 5

- 一旦進入同步與帳號，系統複雜度會明顯上升。
- 這不該在核心價值尚未穩定前就做。
- 如果目前主要是自己和朋友使用，這一階段先不追求最高規格隱私，而是先做到可信任的最低標準。

### 進入條件

只有在以下條件成立時才建議進入：

- 你自己已連續使用數週
- 使用情境真的需要跨裝置

---

## Phase 6：隱私與資料保護升級

目標：當產品要從自己 / 朋友測試，走向更公開或更長期使用時，補齊正式的隱私架構。

### 建議功能

1. 明文資料風險評估
   - server 是否能看到明文健康資料
   - 哪些角色可以接觸資料

2. 更高等級的資料保護
   - client-side encryption 可行性評估
   - 加密金鑰管理策略
   - 更細的權限控制

3. 資料治理
   - 資料保留與刪除策略
   - 匯出 / 可攜性政策
   - 審計與異常存取紀錄

4. 媒體與 AI 隱私
   - 照片 / 語音是否上傳第三方
   - 原始媒體是否保留
   - 使用者可否刪除原始媒體

### 為什麼這是 Phase 6

- 這些會顯著提高系統複雜度與維運責任。
- 如果現在主要是自己和朋友使用，先做到最低安全底線比較務實。
- 真正要公開擴大使用時，再把隱私升級做完整才合理。

---

## 明確保留到更後面的項目

這些可以記著，但目前不建議排進接下來幾個 phase：

- 完整品牌食物資料庫
- 條碼掃描
- 社群與排行榜
- 教練 / 營養師協作
- Apple Health / Google Fit 深度整合
- 完整訓練課表系統
- 最高規格隱私架構

---

## 建議實際順序

如果你要一個最務實的後續開發順序，我建議是：

1. `Phase 2`
   - export / import
   - 複製昨天 / 最近餐點
   - 模板排序
   - 回顧可讀性強化

2. `Phase 3`
   - adaptive target
   - weekly insight v2
   - 自然語言週摘要

3. `Phase 4`
   - 語音記餐
   - 照片記餐
   - 歷史餐點搜尋

4. `Phase 5`
   - 帳號
   - 雲端同步
   - native mobile 評估

5. `Phase 6`
   - 隱私與資料保護升級

---

## 最後判斷

如果依照先前討論的產品方向，**下一個最合理的 phase 是 Phase 2，不是直接跳到 AI 記餐或同步**。

原因：

- Phase 2 補的是現在這個 app 真正會讓人卡住的地方
- 它能提高留存與可用性
- 也不會破壞目前簡潔的產品形狀

一句話總結：

> Phase 2 先讓它變成「能連續用幾週的 app」，Phase 5 再讓它變成「自己和朋友也能同步使用的 app」，Phase 6 才補齊更完整的隱私升級。
