# 後續 Phase Roadmap：Health Track

依照先前 `/office-hours`、`/plan-eng-review`、`/plan-design-review` 與整合 review 的結論整理。

這份文件不是重新發明需求，而是把「MVP 之後該做什麼」收斂成可執行的後續 phases。

## 目前 Phase 1 已完成的核心

目前第一版已經完成的核心回饋迴路是：

1. 個人化目標設定
2. 今日儀表板
3. 快速記餐
4. 常吃餐點模板
5. 體重輸入
6. 週 / 月趨勢回顧
7. Local-first PWA-lite
8. Demo data、基本測試與部署基礎

所以後續 phase 的目標，不是把同一件事做更大，而是補上真正能提升使用價值的下一層能力。

## Review 結論

### P1：後續 phase 不應該先做的事

以下項目雖然常見，但不該優先進入下一階段：

- 完整食物資料庫
- 條碼掃描
- 社群功能
- 食譜 / 餐點推薦
- 原生 iOS / Android app
- App Store / Play Store 發佈
- 醫療級建議

原因：
- 這些都會快速擴大 scope，但不會優先強化你目前最有差異化的產品價值。
- 你的核心仍然是「低摩擦記錄 + 飲食與體重的回饋迴路」。

### P1：最值得優先補的方向

真正應該進入下一個 phase 的，是這四類：

1. 資料可攜性
2. 更聰明的回顧與建議
3. 更低摩擦的記錄流程
4. 長期使用所需的同步與平台延伸

---

## Phase 2：讓第一版真的可長期使用

目標：補齊現在最明顯的使用缺口，讓 app 不只是能 demo，而是可以連續用幾週。

### 建議功能

1. `export / import`
   - JSON 匯出
   - JSON 匯入
   - 匯入前格式驗證
   - 匯入覆蓋或合併策略

2. 強化記餐速度
   - 複製昨天餐點
   - 複製最近一筆餐點
   - 模板排序：最近使用 / 最常使用
   - 今天頁直接快速新增常吃餐點

3. 強化回顧可讀性
   - 記錄完整度提示
   - 缺少資料時的保守提示
   - 蛋白質是否明顯不足
   - 熱量與體重方向是否一致

4. 目標與歷史
   - 保留目標變更歷史
   - 回顧時顯示該期間使用的是哪組目標

### 為什麼這是 Phase 2

- `export / import` 是 local-first app 最直接的缺口。
- 快速複製與模板優化能直接提升「30 秒記一餐」這個核心承諾。
- 回顧可讀性強化，能把現有圖表變成更實用的判斷工具。

### 完成標準

- 使用者可以備份與還原自己的資料
- 記錄常吃餐點不需要每次重填
- 回顧畫面不只顯示數字，也能判斷「資料夠不夠看」

---

## Phase 3：把 app 從 tracker 推進到 coach

目標：從單純顯示結果，進一步提供保守但有用的調整建議。

### 建議功能

1. `adaptive target`
   - 根據數週體重趨勢調整下週熱量建議
   - 維持保守規則，不做過度自信推論

2. `weekly insight v2`
   - 整合熱量偏差、蛋白質、記錄完整度、體重方向
   - 產出更具體的下週建議

3. `自然語言週摘要`
   - 例如：
     - 這週平均攝取接近建議量
     - 體重比上月同期下降
     - 蛋白質穩定，但高熱量餐集中在週末

4. `目標微調入口`
   - 使用者可接受或忽略系統建議
   - 建議不直接覆蓋目標，而是先顯示建議值

### 為什麼這是 Phase 3

- 這是你產品和一般 calorie tracker 最可能拉開差異的地方。
- 但它需要先有足夠歷史資料、足夠穩定的記錄習慣，否則很容易變成亂猜。

### 完成標準

- 使用者看完回顧後，知道下週要不要維持、減少或增加攝取
- 建議文案保持保守，不像醫療建議

---

## Phase 4：降低記錄摩擦的 10x 功能

目標：把輸入速度再往前推一級，但只在核心 loop 已證明有用後才做。

### 建議功能

1. 語音記餐
   - 口語輸入餐點與營養資訊
   - 轉成結構化表單後再讓使用者確認

2. 照片記餐
   - 拍照後產生初始估算
   - 使用者手動確認 / 修正

3. 長按快速新增
   - 從今天頁或模板列表快速插入常用餐點

4. 更好的餐點搜尋
   - 不是大型公共資料庫
   - 先從自己的模板與歷史餐點搜尋開始

### 為什麼這是 Phase 4

- 語音和照片很吸引人，但它們不是第一個瓶頸。
- 如果沒有先證明「使用者真的會回來看回顧」，再快的輸入也只是在優化一個未被證明的流程。

---

## Phase 5：跨裝置與平台延伸

目標：把個人 side project 推向較完整產品。

### 建議功能

1. 帳號系統
2. 雲端同步
3. 多裝置一致性
4. 衝突處理策略
5. 資料 migration
6. Native mobile 評估

### 為什麼這是 Phase 5

- 一旦進入同步與帳號，系統複雜度會明顯上升。
- 這不該在核心價值尚未穩定前就做。

### 進入條件

只有在以下條件成立時才建議進入：

- 你自己已連續使用數週
- `export / import` 已不夠用
- 使用情境真的需要跨裝置

---

## 明確保留到更後面的項目

這些可以記著，但目前不建議排進接下來幾個 phase：

- 完整品牌食物資料庫
- 條碼掃描
- 社群與排行榜
- 教練 / 營養師協作
- Apple Health / Google Fit 深度整合
- 完整訓練課表系統

---

## 建議實際順序

如果你要一個最務實的後續開發順序，我建議是：

1. `Phase 2`
   - export / import
   - 複製昨天 / 最近餐點
   - 模板排序
   - 回顧可讀性強化

2. `Phase 3`
   - adaptive target
   - weekly insight v2
   - 自然語言週摘要

3. `Phase 4`
   - 語音記餐
   - 照片記餐
   - 歷史餐點搜尋

4. `Phase 5`
   - 帳號
   - 雲端同步
   - native mobile 評估

---

## 最後判斷

如果依照先前討論的產品方向，**下一個最合理的 phase 是 Phase 2，不是直接跳到 AI 記餐或同步**。

原因：

- Phase 2 補的是現在這個 app 真正會讓人卡住的地方
- 它能提高留存與可用性
- 也不會破壞目前簡潔的產品形狀

一句話總結：

> Phase 2 先讓它變成「能連續用幾週的 app」，Phase 3 再讓它變成「會給你方向的 app」。
