## 產品設計主張

這是 app UI，不是 landing page。

設計感受應該是「冷靜、可信、每天能快速使用的減脂控制台」：

- 速度優先：主要行動永遠是記錄一餐。
- 誠實：數字是估算與趨勢，不是醫療結論。
- 本機優先：離線 / 本機儲存狀態要清楚，但不要嚇人。
- 每週迴路：週回顧是產品差異化，不是藏起來的報表。

避免：
- generic dashboard-card 拼貼。
- 紫色或藍紫色漸層。
- 裝飾 blob、emoji、圓形 icon 裝飾、generic SaaS 起手式。
- `Welcome to...` 這類行銷文案。
- 像試算表的密集 UI。

## 資訊架構

手機端主要導覽使用底部導覽列：

```text
┌────────────────────────────────────┐
│ 今日                               │
│ 記錄                               │
│ 週回顧                             │
│ 設定                               │
└────────────────────────────────────┘
```

Routes：

```text
/setup          第一次使用的目標設定
/today          今日儀表板
/meals/new      快速記餐
/templates      常吃餐點模板
/weigh-ins      每週體重輸入
/weekly         每週回顧
/settings       設定、PWA / 離線 / 匯出說明
```

建議導覽行為：

- `/setup` 是專注的 onboarding flow，完成目標設定前不顯示底部導覽。
- `/today`、`/meals/new`、`/weekly`、`/settings` 是底部導覽 tab。
- `/templates` 從 `/meals/new` 進入，不一定要是主 tab。
- `/weigh-ins` 可以是 `/weekly` 裡的操作，不一定要是主 tab。

主要旅程：

```text
第一次使用
  │
  ▼
/setup
  │ 儲存目標
  ▼
/today
  │ 點「記一餐」
  ▼
/meals/new
  │ 儲存餐點或模板
  ▼
/today
  │ 週末回顧
  ▼
/weekly
  │ 新增體重
  ▼
/weekly 更新攝取量與體重趨勢
```

## 畫面規格

### `/setup` 目標設定

目標：2 分鐘內完成。

層級：

1. 標題：`設定你的減脂基準`
2. 輔助文字：`用來估算每日熱量與三大營養素。之後可以調整。`
3. 表單欄位：
   - 身高 cm
   - 目前體重 kg
   - 目標體重 kg
   - 年齡
   - 性別
   - 活動量
   - 目標方向：減脂 / 維持 / 增重
4. 結果預覽：
   - 每日建議熱量
   - 蛋白質 / 脂肪 / 碳水
5. 主要行動：`開始追蹤`

設計說明：
- 有效輸入後，結果預覽應即時更新。
- 欄位尚未填完整時，顯示安靜的 placeholder：`填完資料後會顯示建議目標。`
- 放一段小字：`這是粗略估算，不是醫療建議。`

### `/today` 今日儀表板

目標：使用者打開 app 5 秒內就知道現在狀態與下一步。

層級：

1. 日期與本機狀態：`今天 · 本機儲存`
2. 主要數字：`還可攝取 520 kcal` 或 `已超過 180 kcal`
3. 三大營養素進度列：
   - 蛋白質
   - 脂肪
   - 碳水
4. 主要行動：`記一餐`
5. 最近餐點列表
6. 本週預覽：`本週平均 1,920 / 2,100 kcal · 體重 -0.3 kg`

版面草圖：

```text
今天 · 本機儲存

還可攝取
520 kcal
目標 2,100 · 已吃 1,580

[記一餐]

蛋白質  82 / 130g  ━━━━━━━
脂肪    42 / 58g   ━━━━━
碳水    155 / 230g ━━━━━━

最近紀錄
雞胸便當       620 kcal
無糖豆漿       120 kcal

本週狀態
平均攝取低於目標 180 kcal，體重 -0.3 kg
```

設計說明：
- 熱量剩餘數字是視覺焦點。
- Macro bars 要可讀，但層級低於熱量狀態。
- 最近餐點用 plain rows，不做裝飾性卡片。
- 本週預覽值得佔版面，因為它強化產品回饋迴路。

### `/meals/new` 快速記餐

目標：30 秒內記錄常吃餐點。

層級：

1. 標題：`記一餐`
2. 常吃模板列：`常吃`
3. 表單：
   - 餐點名稱
   - 熱量
   - 蛋白質
   - 脂肪
   - 碳水
   - 用餐時間
4. Toggle：`儲存成常吃餐點`
5. 主要行動：`儲存`

版面草圖：

```text
記一餐

常吃
[雞胸便當] [茶葉蛋 + 地瓜] [無糖豆漿]

餐點名稱
熱量 kcal

蛋白質 g    脂肪 g
碳水 g      用餐時間

[ ] 儲存成常吃餐點

[儲存]
```

設計說明：
- 常吃模板應該是 one-tap chips，並有足夠高度給拇指點擊。
- 點模板後先帶入表單，不要立刻儲存，避免誤記。
- 未來可以加 long press 快速新增，不放 MVP。

### `/templates` 常吃餐點

目標：不用建立完整食物資料庫，也能讓重複記錄變快。

層級：

1. 標題：`常吃餐點`
2. 沒有模板時的空狀態：
   - `還沒有常吃餐點`
   - `記餐時勾選「儲存成常吃餐點」，下次就能快速帶入。`
   - 行動：`去記一餐`
3. 模板列：
   - 名稱
   - 熱量與三大營養素摘要
   - 行動：`帶入`

設計說明：
- 模板列應該密集但好掃描。
- 列表不要顯示太多 macro 細節，完整資訊放在編輯模式。

### `/weekly` 每週回顧

目標：回答「這週吃法有沒有讓體重往目標方向移動」。

層級：

1. 標題：`本週回顧`
2. 週區間選擇器
3. 洞察摘要：
   - `本週平均 1,920 kcal，比目標少 180 kcal`
   - `體重 -0.3 kg`
   - `下週建議：先維持`
4. 攝取量 vs 目標比較
5. 體重趨勢
6. 新增 / 更新體重行動

版面草圖：

```text
本週回顧
4/6 - 4/12

這週吃法有效嗎？
平均 1,920 / 2,100 kcal
體重 -0.3 kg
下週先維持

攝取 vs 目標
熱量  -180 kcal/day
蛋白質 -18 g/day
脂肪   +4 g/day
碳水   -42 g/day

體重
72.4 kg → 72.1 kg

[更新本週體重]
```

設計說明：
- 不要過度宣稱建議。使用 `可能代表`、`先維持`、`觀察一週` 這類語氣。
- 如果沒有體重紀錄，仍顯示攝取摘要，並給清楚的體重輸入 CTA。
- 如果只有一筆體重紀錄，不要宣稱趨勢。

### `/settings` 設定

目標：讓 PWA / local-first 限制清楚可見。

建議區塊：

- 目標：編輯目標設定。
- 資料：第一版先不做匯出 / 匯入，之後需要資料可攜性時再談。
- 儲存：`資料儲存在這台裝置。清除瀏覽器資料可能會移除紀錄。`
- 離線：`安裝到主畫面後，可離線開啟已載入的 app。`

這個畫面負責建立信任。不要隱藏 local-only 限制。

| 功能 | 載入中 | 空狀態 | 錯誤 | 成功 | 部分資料 |
|---|---|---|---|---|---|
| Setup | 表單可直接顯示，不需要 skeleton | N/A | 欄位旁顯示錯誤 | 顯示目標預覽，再導向 Today | 未填完整時顯示 muted preview placeholder |
| Today | IndexedDB 開啟較慢時顯示短載入文字 | `今天還沒有紀錄` + `記一餐` | `讀取本機資料失敗` + retry | 儲存後更新總量 | 有 goal 但沒有 meals 時顯示 0 狀態 |

| Templates | DB 慢時顯示列表載入狀態 | 溫和空狀態 + `去記一餐` | Retry + 保留導覽 | 選擇模板後帶入記餐表單 | 刪除模板不影響舊餐點 |
| Weekly | 只在圖表區顯示 skeleton | 沒有 meals 時說明需要先記錄 | 圖表 / 資料錯誤時可 retry | 顯示洞察摘要 | 沒有 weigh-in 或只有一筆 weigh-in 時顯示有限解讀 |

## 使用者旅程與情緒弧線

| 步驟 | 使用者動作 | 可能感受 | 設計回應 |
|---|---|---|---|
| 第一次打開 | 開始 setup | 不確定會不會很麻煩 | 表單短、估算預覽清楚、沒有行銷文案 |
| 第一餐 | 記錄餐點 | 想要快 | 常吃模板明顯、數字欄位分組、只有一個主儲存行動 |
| 中午查看 | 打開 Today | 想知道現在狀態 | 一個大的熱量狀態，macro details 次要 |
| 週末回顧 | 打開 Weekly | 想知道努力有沒有用 | 圖表前先給白話摘要 |
| 離線使用 | 打開已安裝 PWA | 想要可靠 | app shell 可開啟，本機狀態清楚 |
| 長期使用 | 重複記餐 | 想少打字 | 模板與最近餐點降低輸入量 |

時間尺度：

- 前 5 秒：使用者看到剩餘熱量與 `記一餐`。
- 前 5 分鐘：使用者能完成目標設定並記錄至少一餐。
- 長期：每週回顧讓重複記錄變得有意義。

## 視覺系統

分類：App UI。

建議方向：

- 字體：中文 UI 使用 `Noto Sans TC`，fallback 在指定字體之後才使用 `ui-sans-serif`。
- 背景：接近白色的中性色，不用 beige / cream。
- 主要文字：接近黑色。
- 次要文字：中性灰。
- Accent：用單一 green 或 teal 表示健康進度。避免紫色 / 藍紫色。
- Warning：接近上限用 amber，真正超標才用 red。
- Radius：按鈕與 panel 最大 8px。
- 間距尺度：4 / 8 / 12 / 16 / 24 / 32。
- 觸控目標：至少 44px。
- Cards：少用。主要熱量狀態可以是 panel；最近餐點應該是 rows。

建議 token：

```css
:root {
  --color-bg: #f7f8f6;
  --color-surface: #ffffff;
  --color-text: #111827;
  --color-muted: #6b7280;
  --color-border: #d9ded6;
  --color-accent: #0f766e;
  --color-warning: #b7791f;
  --color-danger: #b91c1c;
  --radius-md: 8px;
  --space-1: 4px;
  --space-2: 8px;
  --space-3: 12px;
  --space-4: 16px;
  --space-6: 24px;
  --space-8: 32px;
}
```

Typography：

- 頁面標題：24px / 32px, 700。
- 主要數字：48px / 52px, 700，使用 tabular numerals。
- 區塊標題：16px / 24px, 700。
- 內文：15px / 22px。
- Caption：13px / 18px。

## Responsive 與 PWA 行為

Mobile first：

- 基準寬度：375-430px。
- 底部導覽固定在 safe area 內。
- 主要行動 `記一餐` 要在拇指容易點到的位置。
- 避免 desktop-first sidebar。

Tablet：

- 內容最大寬度約 720px。
- 每週回顧可以並排顯示攝取比較與體重趨勢。

Desktop：

- app 置中在 app-width shell，最大寬度 960px。
- 不要把它變成 marketing landing page。

PWA：

- standalone mode 要能自然隱藏 browser chrome。
- 尊重 safe-area insets。
- 離線提示要低調：`離線模式，本機資料仍可使用。`
- Settings 必須說明 local-only storage。
- service worker 更新策略使用 `autoUpdate`，MVP 不做手動更新提示。
- PWA icon 方向：白底、深綠 / teal 線條、抽象進度環 + 小折線。不要用食物 emoji、醫療十字或複雜插畫。
- icon 檔案至少包含 192x192 PNG、512x512 PNG、maskable 512x512 PNG。

## Accessibility

- 所有互動元素至少 44px 高。
- Inputs 要有可見 label，不只用 placeholder。
- 三大營養素進度列需要文字數值，不能只靠顏色。
- 超標狀態要用文字加顏色。
- 底部導覽 label 必須保留。
- 使用語意 landmarks：`main`、`nav`、`form`。
- 圖表前要有文字摘要。
- Keyboard order 跟視覺順序一致。
- Color contrast 目標：WCAG AA。

## 文案規則

使用工具型語言，不使用行銷語言。

好文案：
- `還可攝取 520 kcal`
- `本週平均比目標少 180 kcal`
- `先觀察一週`
- `資料儲存在這台裝置`

避免：
- `Unlock your health journey`
- `AI-powered health companion`
- `完美掌控你的健康`
- `你失敗了` 或讓人有罪惡感的警告

健康建議文案要保守：

- 使用 `可能代表`、`建議觀察`、`先維持`。
- 不要說 `你應該`、`你必須`、`保證減重`。

## 不在範圍內

- 完整設計系統 / 品牌指南：之後有用，但第一版 UI 實作前不是必要。
- 高保真生成 mockup：目前被 gstack designer 缺少 OpenAI API key 擋住。
- Marketing landing page：這是 app UI。
- Dark mode：等核心 UI 完成後再做。
- 動畫系統：先延後。第一版只做最小互動回饋。
- Push notification：MVP 外。
