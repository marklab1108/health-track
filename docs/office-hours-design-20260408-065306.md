## 問題定義

做一個健康管理 app，服務正在減脂、想控制飲食，但不想把食物紀錄變成麻煩例行公事的人。

這個產品的核心不是「小型 MyFitnessPal」，而是一個緊湊的回饋迴路：

1. 設定個人目標。
2. 大約 30 秒內記完一餐。
3. 看到今日熱量與三大營養素是否符合目標。
4. 回顧本週平均攝取量與體重變化。
5. 用這個回饋調整下週的飲食行為。

## 酷的地方

真正的亮點是把每週體重變化和飲食紀錄放在同一個視圖裡。它不只是說「你今天熱量超標了」，而是幫使用者回答：

> 我這週的吃法，有沒有真的讓體重往目標前進？

第二個產品約束是速度。如果記一餐超過 30 秒，這個 app 就失去意義。

## 目標使用者

- 正在減脂的人。
- 想控制飲食，但懶得維護詳細食物紀錄的人。
- 第一個具體使用者：開發者自己。

## 約束

- 這是 side project / learning project，所以範圍應該能練到有價值的 app 開發模式，但不能膨脹。
- 第一版不要維護完整食物資料庫。
- 第一版不要做條碼掃描、照片辨識、語音輸入、社群功能、餐點推薦系統。
- app 不應該假裝提供醫療建議。它應該把計算和趨勢呈現成規劃輔助。

## 前提

1. 第一版不做完整食物資料庫。真正有用的切入點是快速記錄加每週回饋，不是食物搜尋覆蓋率。
2. 每週回顧應該是核心畫面，不是附屬圖表。這是它和一般營養資料維護 app 的主要差異。
3. 熱量與三大營養素目標可以先用靜態公式，但介面要預留未來根據體重趨勢調整下週建議的路徑。
4. 語音和照片記餐是 10x 未來功能，不是 MVP 必要功能。常吃餐點模板才是第一版更好的速度解法。

## 評估過的方案

### 方案 A：最小可用回饋迴路

先做目標設定、今日儀表板、快速記餐、每週回顧。餐點用手動輸入，搭配常吃餐點模板來降低摩擦。

投入：中
風險：低

優點：
- 最快做出使用者真的會用的版本。
- 可以驗證核心回饋迴路。
- 避免掉進 MyFitnessPal 的食物資料庫坑。

缺點：
- 使用者不能搜尋大型食物資料庫。
- 手動輸入營養素仍然有一點摩擦。
- 常吃餐點模板要等使用者建立常用餐點後，價值才會變明顯。

### 方案 B：先做自動調整教練

第一版就把每週調整建議放在中心。使用者記錄飲食和體重後，app 產生每週分析，並調整下週熱量目標。

投入：大
風險：中

優點：
- 最接近長期 10x 版本。
- 和基本追蹤器相比，差異化更明顯。
- 有 2-4 週資料後會更有價值。

缺點：
- 第一週資料不足，產品價值會比較弱。
- 如果調整規則設計不好，會顯得隨機或過度自信。
- 文案需要小心，避免聽起來像醫療建議。

### 方案 C：超快速記錄原型

先專注在快速記錄介面：常吃餐點、複製昨天餐點、一鍵加入。每週回顧先做簡單版。

投入：小
風險：低

優點：
- 最快能做出 demo。
- 直接解決「懶得記錄」這個問題。
- 第一版實作範圍最小。

缺點：
- 會弱化每週體重和飲食洞察。
- 容易變成一般熱量輸入 app。
- 會延後最有差異化的價值。

## 推薦方案

選擇方案 A：最小可用回饋迴路。

它足夠完整，可以真的有用，同時避開最大的 scope trap。第一版應該交付完整回饋迴路，而不是只做一個孤立畫面：

1. 個人化目標設定。
2. 每日熱量與三大營養素目標計算。
3. 今日儀表板。
4. 快速餐點輸入。
5. 常吃餐點模板。
6. 每週體重輸入。
7. 每週回顧，比較平均攝取量、建議攝取量和體重變化。

## MVP 範圍

### 目標設定

輸入：
- 身高
- 目前體重
- 目標體重
- 目標方向，例如減脂 / 維持 / 增重

輸出：
- 每日建議熱量
- 每日建議蛋白質
- 每日建議脂肪
- 每日建議碳水化合物

第一版公式可以刻意簡單。重點是要能在 UI 裡說清楚，也要容易在之後替換。

### 今日儀表板

顯示：
- 已攝取熱量 vs 每日目標
- 已攝取蛋白質 vs 每日目標
- 已攝取脂肪 vs 每日目標
- 已攝取碳水化合物 vs 每日目標
- 剩餘量或超標量
- 快速記餐入口

儀表板應該像控制台，不像試算表。

### 餐點紀錄

必要欄位：
- 餐點名稱
- 熱量
- 蛋白質
- 脂肪
- 碳水化合物
- 日期 / 用餐時間

快速路徑：
- 從常吃餐點模板加入
- 複製最近吃過的餐點
- 將目前餐點儲存成模板

### 每週體重追蹤

允許使用者每週輸入一次體重。

顯示：
- 每週體重趨勢
- 和上一週相比的變化
- 方向是否符合使用者目標

### 每週回顧

顯示：
- 本週平均每日熱量
- 本週平均每日蛋白質、脂肪、碳水化合物
- 建議每日目標
- 平均攝取量和建議量的差距
- 本週體重變化

核心讀法：
- 「這週你平均每天攝取 X 熱量，目標是 Y。」
- 「體重變化是 Z。」
- 「這代表下週可以維持 / 稍微減少 / 稍微增加。」

最後一句在 MVP 裡要保守。它可以是規則式判斷，不需要 AI。

## 10x 方向

後續功能：
- 常吃餐點模板與一鍵加入。
- 根據體重趨勢自動調整下週熱量建議。
- 語音或照片記餐。
- 每週自然語言分析飲食與體重。
- 把健身訓練量納入每週回顧。

可能的順序：

1. 常吃餐點模板。
2. 自動調整下週目標建議。
3. 健身訓練量。
4. 自然語言每週摘要。
5. 語音或照片記餐。

照片和語音很酷，但不是第一個瓶頸。第一個瓶頸是使用者能不能在低摩擦下建立記錄習慣。

## 未決問題

1. 第一版熱量與三大營養素目標要用哪個公式？
2. 使用者資料第一版已選擇 local-first；後續是否需要匯出 / 匯入？
3. 第一版已選擇 mobile-first PWA-lite；後續是否需要 native mobile？
4. 最小可用的每週調整規則是什麼？
5. 體重要每週輸入一個值，還是每天輸入再整理成週趨勢？

## 成功標準

- 使用者可以在 2 分鐘內完成目標設定。
- 建立餐點模板後，使用者可以在 30 秒內記錄一個常吃餐點。
- 今日儀表板能清楚顯示使用者是否接近或超過今日熱量與三大營養素目標。
- 每週回顧能回答平均攝取量是否超過建議量。
- 每週回顧能顯示體重是否往目標方向移動。
- v1 不建立也不依賴大型食物資料庫。

## 發佈計畫

作為學習型 side project，建議先做 mobile-first PWA-lite web app。

建議發佈方式：
- 第一個垂直切片可用後，部署到公開 URL。
- 依照選定技術棧，先用本機資料或輕量 hosted database。
- 加一個簡單的種子資料 / demo 模式，讓每週回顧在使用者還沒有完整一週資料前也能展示。

CI/CD 可以保持簡單：
- PR 上跑測試和型別檢查。
- 有部署流程後，合併到 main 就自動部署。

## 我注意到的事

- 你沒有停在「記錄熱量」。你明確想把每週體重變化和飲食紀錄連起來。這是產品洞察。
- 你在意速度：「30 秒內記完一餐」。這是好約束，能防止 app 變成比較漂亮的試算表。
- 你說想變成 MyFitnessPal，但真正的切入點更窄也更個人：一個為減脂設計的輕量追蹤器，而不是完整食物資料庫產品。
