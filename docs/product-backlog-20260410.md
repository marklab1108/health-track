# Product Backlog：Health Track

依照目前的 roadmap，這份 backlog 把後續工作拆成可以直接開 issue 的粒度。

排序原則：

1. 先做真正會提升留存與可用性的項目
2. 再做讓產品更聰明的項目
3. 最後才做高成本、高吸引力但不是第一瓶頸的功能

## Now：Phase 2 Backlog

### P2.1 更快的記錄流程

- [ ] 複製昨天餐點

## 建議 issue 開法

如果你接下來要在 GitHub 開 issue，我建議先只開這 6 個：

1. `P2.1 複製昨天 / 最近餐點`
2. `P2.2 模板排序與快速加入`
3. `P2.3 回顧可讀性強化`
4. `P2.4 目標歷史`
5. `P2.5 空狀態 / 錯誤狀態整理`
6. `P2.6 demo data 與測試流程整理`

這樣比較符合現在產品成熟度，也最容易持續推進。
