# DESIGN.md

這份文件整合目前 `Health Track` 專案兩次 `/design-consultation` 的結論，作為後續 UI/UX 設計與實作的基準文件。

目標不是做出花俏介面，而是建立一套一致、好維護、能支撐日常使用的設計規範。

## 1. 產品設計定位

`Health Track` 是一個 mobile-first、local-first 的健康追蹤 app。

設計核心不是「展示很多健康數據」，而是讓使用者能在最短時間內完成這三件事：

1. 快速記錄今天吃了什麼
2. 立刻知道今天還能怎麼調整
3. 在週回顧看懂飲食和體重的關係

所以 UI 要優先服務：

- 高頻操作
- 清楚結論
- 穩定回饋
- 低認知負擔

不追求：

- SaaS dashboard 感
- 管理後台感
- 資訊密度過高的健康控制面板

## 2. 核心設計原則

### 2.1 Dashboard 要有主次，不是資訊平均分配

每個畫面都要有明確主軸。

- `/today` 的主軸是：今天剩多少、接下來做什麼
- `/log` 的主軸是：最快完成一筆餐點輸入
- `/weekly` 的主軸是：這週的吃法有沒有用
- `/setup` 的主軸是：建立或調整目前目標

不要把所有資訊都做成同樣重要的卡片。

### 2.2 先給結論，再給數據

使用者不是來看 chart library，而是來做判斷。

所以：

- 回顧頁先顯示結論，再顯示摘要卡，再顯示圖表
- Today 頁先顯示剩餘熱量與當前狀態，再顯示操作入口
- 缺資料時先提示「目前不能下結論」，而不是照樣塞圖

### 2.3 快速操作要集中，不要分散在多處重複出現

高頻操作應集中在一個合理區塊，不應在多個畫面重複放相同入口。

因此：

- `常用模板` 以 `/log` 為主，不在 `/today` 重複做一套
- `/today` 只保留真正高頻的快捷操作：
  - `記一餐`
  - `複製昨天`
  - `複製最近一筆`

### 2.4 使用者不需要看到工程語言

像 `V2`、版本號、資料結構概念，不應直接出現在主要使用流程。

使用者應看到的是：

- `目前目標`
- `依目前目標計算`
- `目標已於 YYYY/MM/DD 更新`

不要讓 UI 把工程內部概念直接暴露給使用者。

### 2.5 視覺語言要收斂

整個 app 的主要 UI 元件應收斂為少數幾種：

1. `Summary card`
   - 顯示單一重要數值與簡短上下文
2. `Action row / action card`
   - 承載可直接操作的高頻行為
3. `List row`
   - 顯示可掃描的餐點、歷史、搜尋結果
4. `Status badge`
   - 只表達狀態，不承擔主要資訊

避免同一種資料有時像 chip、有時像 card、有時像 row，導致畫面節奏混亂。

## 3. 畫面規範

### 3.1 `/today`

#### 角色

高頻控制台。

它不是第二個 `/log`，也不是只有圖表的 dashboard。

#### 正確資訊順序

1. Hero：今日剩餘熱量 / 是否超標
2. Macro summary：蛋白質 / 脂肪 / 碳水進度
3. Quick actions：`記一餐 / 複製昨天 / 複製最近一筆`
4. 今天的餐點列表

#### 明確規範

- `/today` **不放常用模板區塊**
- Quick actions **不要放在最上面**，也不要夾在不自然的位置
- 建議放在 hero 與 macro 之後，餐點列表之前
- `記一餐` 是主 CTA
- `複製昨天`、`複製最近一筆` 是次要 CTA
- 成功回饋要固定位置，不可因 reload 消失

#### 使用者文案方向

- `還能吃 1,240 kcal`
- `今天已記錄 3 餐`
- `已複製昨天的 2 餐`
- `已加入雞胸便當`

短、直接、可立即理解。

### 3.2 `/log`

#### 角色

最快完成輸入的頁面。

#### 設計方向

`/log` 承擔所有「快速選用既有資料」的任務：

- 常用模板
- 最近記錄
- 搜尋最近餐點
- 手動輸入

#### 正確資訊順序

1. 搜尋 / 快速找餐點
2. 常用模板
3. 最近記錄
4. 手動輸入表單

#### 明確規範

- `常用模板` 只留在 `/log`
- 搜尋結果、最近記錄、模板要盡量共用相近的視覺結構
- 不要讓 `/log` 看起來像四個不同模組硬疊在一起
- 手動輸入表單是必要功能，但不是第一視覺重點

### 3.3 `/weekly`

#### 角色

判讀頁，而不是圖表展示頁。

#### 正確資訊順序

1. 主結論
2. 補充說明 / 保守提示
3. 資料摘要卡
4. 圖表
5. 新增體重

#### 明確規範

- 頁首先給一句結論，不要一上來只有平均熱量數字
- `完整度` 可以顯示，但屬於狀態資訊，不應搶過主結論
- 不顯示 `V2` 這種版本字眼
- 可改為：
  - `依目前目標計算`
  - `目標更新於 YYYY/MM/DD`
- `本月體重變化` 是產品核心訊號，重要性高於部分營養摘要卡
- 圖表標題下可有一句輔助說明，幫使用者讀圖

### 3.4 `/setup`

#### 角色

建立或調整目前目標。

#### 明確規範

- 可以有「目前目標」或「最近更新時間」
- 不在主要設定流程暴露版本號
- 歷史感存在即可，不要讓畫面變成設定管理後台

### 3.5 `/settings`

#### 角色

低頻設定與測試工具入口。

#### 明確規範

- 可放 demo data
- 可放 goal history 的輕量資訊
- 不要把工程細節、local-first、PWA-lite 等概念寫給一般使用者看
- 使用者只需要知道與自身操作有關的內容

## 4. 文字與資訊表達規範

### 要用的語氣

- 直接
- 保守
- 不審判使用者
- 不假裝提供醫療判斷

### 避免的語氣

- `你失敗了`
- `你今天吃太多`
- `系統版本 V2`
- `本機優先 / PWA-lite / service worker` 這類工程詞

### 建議文案方向

- `先維持這個節奏`
- `這週資料還不夠完整，先再記幾天`
- `先把蛋白質補齊，再觀察熱量調整`
- `依目前目標計算`

## 5. 視覺系統規範

### 色彩

維持目前的中性底色 + teal/green accent 方向。

- 主色：teal / green
- 警示：orange / red
- 背景：淺灰或白

不要使用：

- 紫色或紫藍漸層
- 裝飾性 blob
- 過度 SaaS 化的炫技配色

### 圓角與元件

- card / button radius 維持 8px
- touch target 至少 44px
- 元件間距應優先讓資訊清楚分組，不要純靠線條堆砌

### 卡片密度

- summary cards 可稍微大一點，承載主要數字
- action cards 必須一眼看出可點
- list rows 要適合快速掃描，不要太厚重

## 6. 後續實作時的決策規則

未來如果 UI 要新增功能，先用這些問題檢查：

1. 這個功能應該放在哪個畫面最自然？
2. 它是高頻操作還是低頻設定？
3. 它會不會讓 `/today` 變成第二個 `/log`？
4. 這段文字是不是把工程語言直接暴露給使用者？
5. 這個新元件能不能用現有的 summary/action/list/badge 語言表達？

如果答案不清楚，先不要加。

## 7. 當前設計結論

目前這版 UI/UX 的方向是對的：

- 功能路徑清楚
- mobile-first 基礎穩定
- 週回顧已開始有產品差異化

但後續要持續收斂這三件事：

1. CTA hierarchy 要更明確
2. `/today`、`/log` 的角色邊界要更乾淨
3. 工程概念不要浮到使用者 UI

後續所有 UI 調整，都應優先遵守這份 `DESIGN.md`。
